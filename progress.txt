## 2026-04-16 - US-003
- Added `src/ceac/navigator.ts` as the central CEAC page-transition abstraction. All DS-160 forward/back/save moves should route through it so navigation is never timing-only and validation failures surface with page context.

Each transition asserts the origin page identity, clicks the correct CEAC nav button, short-circuits on inline validation errors, then polls for the destination identity via the existing `waitForPage`.

Learnings:
- CEAC's `asp:ValidationSummary` / per-field validators stay mounted with `display: none`. Filtering by `isVisible()` is the reliable way to read actual errors â€” `textContent()` alone surfaces stale text.

## 2026-04-16 - US-004
- Added `src/ceac/checkpoints.ts`: Application ID capture + structured checkpoint emission for the DS-160 flow. Legacy worker (`src/index.ts`) intentionally untouched per the US-004 scope note â€” integration lands in US-008.
- `captureApplicationId(page)` is best-effort and non-throwing: it walks every known Application-ID selector, falls back to scanning visible body text against `CEAC_APPLICATION_ID_PATTERN`, and returns `{ applicationId, source, raw? }`. Source is one of `selector` | `body_scan` | `none` so callers can tell "not issued yet" from "capture strategy changed".
- `CeacCheckpoint` is fully serializable: `at` (ISO-8601), `action`, `pageId`, `heading`, `url`, `applicationId`, optional `runId`, optional `details`. Emission goes through a pluggable `CheckpointSink`; `consoleCheckpointSink` (the default) prints one JSON line per record so logs remain greppable and aggregation-friendly until a durable sink is wired in.
- `saveAndCheckpoint(page, { at, sink, runId, knownApplicationId, ... })` wraps `navigator.saveCurrent` and emits a `save` checkpoint â€” the canonical "natural section boundary save" primitive. `recordSectionCheckpoint` and `recordBootstrapCheckpoint` cover the non-save cases (advancing into a new section, and the initial bootstrap baseline).
- `buildCheckpoint(page, { applicationId, ... })` treats the `applicationId` input as a three-state signal: `undefined` = scan the page now, `string` = use the caller's cached value (avoids redundant DOM scans across a run), `null` = explicitly record "not yet issued". This keeps the Application ID capture cost to once per run.
- Exported the full surface from `src/ceac/index.ts`. Typecheck (`npm run type-check` in viza-be/submission-service) passes.
Files changed:
- viza-be/submission-service/src/ceac/checkpoints.ts (new)
- viza-be/submission-service/src/ceac/index.ts (export checkpoint surface)
- prd.json (US-004 passes: true)
- progress.txt (this entry)
Learnings:
- Application ID capture must be non-throwing. CEAC does not render the ID until after the applicant finishes the start/security pages, so an early-page capture attempt is an expected "none" â€” treating it as an error would spam alerts on every run bootstrap.
- A three-state `applicationId` input (`undefined` | `string` | `null`) is worth the API complexity: `undefined` triggers a DOM scan, `string` reuses a cached value, `null` records "confirmed not yet issued". Collapsing to a two-state design would have forced either redundant scans or a loss of the "confirmed absent" signal.
- Checkpoint emission belongs outside the navigator. The navigator already throws structured errors; mixing persistence concerns into it would have made failure-path reasoning harder. A pluggable `CheckpointSink` at the call site keeps the layers clean and lets future stories (US-005 `.dat` artifacts, US-007 screenshots) reuse the same sink abstraction.
- Body-text fallback for Application ID is safe because the `AA[A-Z0-9]{8,10}` pattern is distinctive enough that false positives are effectively impossible on CEAC pages â€” no page content uses that format outside the ID itself.

## 2026-04-16 - US-005
- Added `src/ceac/artifacts.ts`: `.dat` Save-to-File capture + a `RecoveryTracker` that accumulates Application ID, last checkpoint, and the `.dat` artifact across a run, plus `preserveRecoveryOnFailure` for the error path. Legacy worker untouched per the story scope (integration lands in US-008).
- `captureDatArtifact(page, { outputDir, ... })` clicks CEAC's Save-to-File button, awaits the Playwright `download` event (listener registered BEFORE the click so we don't race CEAC's fast response), saves the `.dat` to disk via `download.saveAs`, and returns a fully-serializable `DatArtifact` (path, suggestedFilename, sizeBytes, capturedAt, sourcePageId, applicationId). Missing button / failed click / missing download each raise a structured `NavigationError` with `action: "save_to_file"` context.
- `RecoveryTracker` implements the existing `CheckpointSink` interface so callers pass it straight into `saveAndCheckpoint` / `recordSectionCheckpoint` / `captureDatAndCheckpoint`. Every checkpoint updates `lastCheckpoint`; any non-null `applicationId` on a checkpoint is promoted to the tracker-wide value (CEAC surfacing the ID is authoritative). `attachDatArtifact` stores the newest `.dat` (resume flows should use the freshest file) and promotes its `applicationId` as well.
- `captureDatAndCheckpoint` is the canonical "Save-to-File + checkpoint" primitive â€” symmetric with US-004's `saveAndCheckpoint`. Tracker precedence: when supplied, the tracker is used as the checkpoint sink (it already forwards to its delegate), keeping recovery state in sync with the checkpoint log without double-wiring.
- `preserveRecoveryOnFailure` snapshots the tracker, serializes the error (name/message + optional code/context for `CeacError` subclasses), and emits a final `manual` checkpoint with `reason: "run_failed"` carrying the full recovery payload. Page-aware path uses `buildCheckpoint`; if the page is gone it synthesizes a checkpoint so the snapshot still lands in the sink. Wrapped in try/catch to guarantee the preserve path never masks the original error.
- No changes to `src/ceac/checkpoints.ts`: US-004 already reserved the `save_to_file` action string on the `CeacCheckpointAction` union, so artifacts.ts emits it directly.
- Exported the new surface from `src/ceac/index.ts`. Typecheck (`npm run type-check` in viza-be/submission-service) passes.
Files changed:
- viza-be/submission-service/src/ceac/artifacts.ts (new)
- viza-be/submission-service/src/ceac/index.ts (export artifact/recovery surface)
- prd.json (US-005 passes: true)
- progress.txt (this entry)
Learnings:
- Playwright's `download` event can fire before `click()` resolves for small files â€” register `page.waitForEvent("download", ...)` BEFORE clicking, not after. This is why Playwright's docs show the wait pattern as a Promise-holding variable rather than a sequential call.
- US-004 reserved the `save_to_file` action value on `CeacCheckpointAction` specifically so US-005 wouldn't need to widen the union. That prior decision kept the change set to artifacts.ts + the index barrel â€” exactly the minimally-invasive shape the story's scope note called for.
- Making `RecoveryTracker extends CheckpointSink` (rather than wrapping one) lets callers pass the tracker wherever a sink is accepted. That avoids a second "sink + tracker" wiring everywhere and keeps the Application-ID sync logic in one place. Downside: callers who want both a plain sink AND a tracker must configure the tracker's `delegate` â€” documented explicitly on `CreateRecoveryTrackerOptions`.
- Failure-path preservation must be strictly non-throwing. Wrapping `buildCheckpoint` + `sink.record` in try/catch is the only way to guarantee that preserving recovery state never masks the underlying error. The synthetic-checkpoint branch (when `page` is null) exists because the browser/context may already be closed by the time the caller enters its catch block.

## 2026-04-16 - US-006
- Added `src/ceac/stop-at-sign.ts`: the stop-at-sign-page contract that terminates a CEAC autofill run at the Sign and Submit page without ever entering passport-number signature, CAPTCHA, or clicking final submit. Legacy worker (`src/index.ts`) intentionally untouched â€” integration into the worker result payload lands in US-008.
- Sign-page detection is via **explicit DOM markers**, not heading alone. `readSignPageMarkers` probes four signals â€” heading pattern match, passport-signature input presence, final-submit button presence, CAPTCHA image presence. A positive identification requires the heading AND both control-level markers (signature field + final submit). CAPTCHA presence is reported but not required (CEAC sometimes defers CAPTCHA rendering until signature focus).
- `detectSignAndSubmit(page)` is the non-throwing read-only probe: returns `SignPageIdentity | null`. `assertSignAndSubmit(page)` is the strict variant: throws `UnexpectedPageError` with a `missingRequired` diagnostic list enumerating exactly which markers failed (`heading` / `signature_field` / `final_submit_button`).
- `stopAtSignAndSubmit(page, options)` is the terminal primitive. It asserts identity, emits a new `handoff_ready` checkpoint with the marker snapshot in `details`, and returns a `HandoffReadyOutcome` record. The outcome's `status` field is the literal `"handoff_ready"` (never `"submitted"`) so downstream consumers can type-discriminate to prove no submission occurred. `pageId` is also typed as the literal `"sign_and_submit"`.
- Added `"handoff_ready"` to the `CeacCheckpointAction` union in `checkpoints.ts`. This is a union widening, not a narrowing â€” no existing call sites broke. The semantic separation from `"manual"` matters: handoff-ready is the canonical successful terminal state, not a generic caller-initiated checkpoint.
- Recovery integration: when a `RecoveryTracker` is supplied, the tracker serves as the sink AND is read for Application-ID / last-checkpoint / `.dat` artifact context. The tracker snapshot is re-read after emission so `lastCheckpoint` in the outcome reflects the just-emitted stop. Tracker precedence mirrors US-005's `captureDatAndCheckpoint` (tracker > explicit sink > console default) â€” single-path wiring.
- `isHandoffReadyOutcome(value)` exported as a type guard for consumers receiving an untyped worker result (useful once US-008 wires this into the worker output path).
- Exported the new surface from `src/ceac/index.ts`. Typecheck (`npm run type-check` in viza-be/submission-service) passes.
Files changed:
- viza-be/submission-service/src/ceac/stop-at-sign.ts (new)
- viza-be/submission-service/src/ceac/checkpoints.ts (add `handoff_ready` to `CeacCheckpointAction`)
- viza-be/submission-service/src/ceac/index.ts (export stop-at-sign surface)
- prd.json (US-006 passes: true)
- progress.txt (this entry)
Learnings:
- "Explicit markers" in the US-006 acceptance criteria means going beyond heading-only detection. Heading match is necessary but not sufficient â€” CEAC can render a "Sign and Submit"-titled page in anomalous states (outage fallback, post-submit confirmation variants). Requiring both the passport-signature input AND the final submit button in the DOM is what makes the detection resistant to those edge cases.
- The stop contract's strongest form is structural, not runtime: `stopAtSignAndSubmit` contains no code that could fill the signature field, fill the CAPTCHA, or click final submit. No runtime "readOnly=true" flag, no conditional guard â€” just the absence of those calls. Reviewers verify the contract by grep for `fill(` / `click(` on sign-page selectors, which should return zero hits in the stop-at-sign module.
- `pageId: "sign_and_submit"` typed as a string literal (not `CeacPageId`) on `HandoffReadyOutcome` is intentional: it guarantees at compile time that only the stop path can construct this outcome shape, and lets downstream discriminators rely on the literal without a runtime re-check.
- Tracker snapshot must be read BOTH before (for Application ID / runId context on the checkpoint) AND after (so `lastCheckpoint` in the outcome reflects the stop checkpoint itself). Single-read designs either miss the just-emitted checkpoint or skip the pre-emission context.
- The `captchaPresent` marker is surfaced on the outcome but not required for identification. CEAC's captcha rendering is flaky; requiring it would cause spurious failures on perfectly valid sign pages. The marker IS surfaced so ops artifacts/screenshots (US-007) can correlate "captcha rendered vs not" across runs.
